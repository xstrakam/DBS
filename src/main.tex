\section{Specification}
\label{sec:specification}

\subsection{Problem despription}

\subsection{Actors}

\begin{enumerate}
    \item Visitor (unregistered user)
    \begin{itemize}
        \item Can browse movie catalogue
        \item Can search and filter movies in the catalogue
        \item Can read reviews and view ratings
    \end{itemize}
    \item Registered user
    \begin{itemize}
        \item Has all visitor's rights
        \item Can rate movies
        \item Can write reviews
        \item Can create watchlists and share them amongst other users
    \end{itemize}
    \item Moderator
    \item Administrator
    \begin{itemize}
        \item Has all moderator's rights
        \item Can add new movies into database
        \item Can modify movie attributes
    \end{itemize}
\end{enumerate}

\subsection{Key entities}

\begin{enumerate}
    \item Movie
    \item User
    \item Review
    \item Rating
    \item Watchlist
    \item Genre
    \item Person
\end{enumerate}

\subsection{Business rules and restrictions}

\begin{enumerate}
    \item \textbf{Rating uniqueness}
    One user can rate a movie just once. This also includes that only registered users can rate a movie.
    \begin{itemize}
        \item Integrity constraint: UNIQUE constraint on (\texttt{user\_id}, \texttt{movie\_id}) in Rating table
        \item Integrity constraint: NOT NULL foreign key on \texttt{user\_id} in Rating table
        \item Reason: Prevents duplicate ratings that would skew the average score
    \end{itemize}
    \item \textbf{Movie uniqueness}
    Combination of title and release year must be unique for every movie to prevent duplicate entries.
    \begin{itemize}
        \item Integrity constraint: UNIQUE constraint on (\texttt{title}, \texttt{release\_year}) in Movie table
        \item Reason: The same movie with the same release year cannot be added twice
    \end{itemize}
    \item \textbf{Valid release year}
    Movie release year must be realistic.
    \begin{itemize}
        \item Integrity constraint: CHECK constraint (\texttt{release\_year} >= 1888 AND \texttt{release\_year} <= 2100)
        \item Reason: The first movie was filmed in 1888, the upper limit protects against input errors
    \end{itemize}
\end{enumerate}

\section{Use cases}
\label{sec:use_cases}

\subsection{Searching and filtering movies}

Visitor or registered user filters movies based on multiple criteria (genre, release year, rating, director, actors) and sorts them.

\textbf{Main flow:}

\begin{enumerate}
    \item User opens search tab
    \item System displays available filters
    \item User selects and applies filters
    \item User chooses sorting option
    \item System creates SQL query and applies it onto the database
    \item System shows filtered movies
\end{enumerate}

\textbf{Alternate flows:}

\begin{enumerate}
    \item No results
    \begin{itemize}
        \item No movies match the selected filters
        \item System shows ``no matches'' message
        \item User adjusts the filters
    \end{itemize}
\end{enumerate}

\subsection{Adding a Movie (Administrator)}

An administrator adds a new movie entry to the portal, including its
metadata, genres, and cast.

\textbf{Main flow:}

\begin{enumerate}
    \item Administrator navigates to the movie management section
    \item System displays the movie submission form
    \item Administrator fills in movie details (title, release year,
          runtime, genres, directors, actors, writers)
    \item Administrator submits the form
    \item System validates the input (release year range, unique
          title + year combination, positive runtime)
    \item System inserts a new record into \texttt{Movie}, links genres
          via \texttt{MovieGenre}, and links people via \texttt{MoviePerson}
    \item System confirms successful addition
\end{enumerate}

\textbf{Data layer:}
\begin{itemize}
    \item INSERT into \texttt{Movie} (title, release\_year, runtime\_minutes)
    \item INSERT into \texttt{MovieGenre} for each selected genre
    \item INSERT into \texttt{MoviePerson} for each person with their role
\end{itemize}

\textbf{Alternate flows:}
\begin{enumerate}
    \item Duplicate movie
    \begin{itemize}
        \item A movie with the same title and release year already exists
        \item System rejects the submission and displays an error message
        \item Administrator corrects the entry or cancels
    \end{itemize}
    \item Invalid data
    \begin{itemize}
        \item Release year or runtime fails CHECK constraint
        \item System highlights the invalid fields
        \item Administrator corrects the values and resubmits
    \end{itemize}
\end{enumerate}

\subsection{Rating a Movie}

A registered user submits a numerical rating (1--10) for a movie they
have watched.

\textbf{Main flow:}

\begin{enumerate}
    \item Registered user opens a movie detail page
    \item System checks that the user has not already rated this movie
    \item User selects a score between 1 and 10 and confirms
    \item System inserts a record into \texttt{Rating}
    \item System recalculates \texttt{average\_rating} and increments
          \texttt{total\_ratings} in \texttt{Movie}
    \item Updated average rating is displayed on the movie page
\end{enumerate}

\textbf{Data layer:}
\begin{itemize}
    \item SELECT on \texttt{Rating} to check (user\_id, movie\_id) uniqueness
    \item INSERT into \texttt{Rating} (user\_id, movie\_id, score)
    \item UPDATE \texttt{Movie} SET average\_rating, total\_ratings
\end{itemize}

\textbf{Alternate flows:}
\begin{enumerate}
    \item User already rated this movie
    \begin{itemize}
        \item UNIQUE constraint on (user\_id, movie\_id) prevents duplicate
        \item System offers the option to update the existing rating instead
    \end{itemize}
    \item User is not logged in
    \begin{itemize}
        \item System redirects to the login page
    \end{itemize}
\end{enumerate}

\subsection{Creating and Managing a Watchlist}

A registered user creates a watchlist, adds movies to it, and optionally
shares it publicly.

\textbf{Main flow:}

\begin{enumerate}
    \item Registered user navigates to their profile and selects
          ``Create watchlist''
    \item User enters a name, optional description, and sets visibility
          (public / private)
    \item System inserts a new record into \texttt{Watchlist}
    \item User browses the movie catalogue and adds movies to the watchlist
    \item System inserts records into \texttt{WatchlistMovie}
          (watchlist\_id, movie\_id)
    \item If the watchlist is public, other users can view and browse it
\end{enumerate}

\textbf{Data layer:}
\begin{itemize}
    \item INSERT into \texttt{Watchlist} (user\_id, name, description,
          is\_public)
    \item INSERT into \texttt{WatchlistMovie} (watchlist\_id, movie\_id)
          for each added movie
    \item SELECT on \texttt{Watchlist} WHERE is\_public = TRUE for
          public browsing
\end{itemize}

\textbf{Alternate flows:}
\begin{enumerate}
    \item Movie already in watchlist
    \begin{itemize}
        \item UNIQUE constraint on (watchlist\_id, movie\_id) prevents
              duplicates
        \item System notifies the user that the movie is already on
              the list
    \end{itemize}
    \item User deletes a watchlist
    \begin{itemize}
        \item ON DELETE CASCADE removes all associated
              \texttt{WatchlistMovie} records
    \end{itemize}
\end{enumerate}

\section{Relational Model Design}

\subsection{Overview of tables and attributes}

\subsubsection{User}

\noindent\textbf{Purpose:} Stores information about registered users.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: SERIAL, PRIMARY KEY % v postgreSQL serial == integer auto increment
    \item \texttt{first\_name}: VARCHAR(50), NOT NULL
    \item \texttt{last\_name}: VARCHAR(50), NOT NULL
    \item \texttt{username}: VARCHAR(30), NOT NULL, UNIQUE
    \item \texttt{email}: VARCHAR(254), NOT NULL, UNIQUE
    \item \texttt{password\_hash}: VARCHAR(60), NOT NULL % bcrypt hash
    \item \texttt{role}: ENUM('user', 'admin'), NOT NULL, DEFAULT 'user'
    \item \texttt{created\_at}: TIMESTAMP, NOT NULL, DEFAULT CURRENT\_TIMESTAMP
\end{itemize}

\noindent\textbf{Primary key:} Surrogate key \texttt{id}. A surrogate key is preferred over username or email because both, although unique, might be changed in the future.

\noindent\textbf{Foreign keys:} None.

\subsubsection{Movie}

\noindent\textbf{Purpose:} Stores information about movies.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: SERIAL, PRIMARY KEY
    \item \texttt{title}: VARCHAR(200), NOT NULL
    \item \texttt{release\_year}: INTEGER, NOT NULL, CHECK (release\_year >= 1888 AND release\_year <= 2030)
    \item \texttt{runtime\_minutes}: INTEGER, CHECK (runtime\_minutes > 0)
    \item \texttt{average\_rating}: NUMERIC(3,1), DEFAULT NULL, CHECK (average\_rating >= 0 AND average\_rating <= 10)
    \item \texttt{total\_ratings}: INTEGER, NOT NULL, CHECK (total\_ratings >= 0), DEFAULT 0
    \item \texttt{created\_at}: TIMESTAMP, NOT NULL, DEFAULT CURRENT\_TIMESTAMP
\end{itemize}

\noindent\textbf{Primary key:} Surrogate key \texttt{id}. An artificial key is preferred over the combination of title and release year, because title can be lenghty and id is more efficient for indexing.

\noindent\textbf{Foreign keys:} None.

\subsubsection{Person}

\noindent\textbf{Purpose:} Stores information about people involved in movies (directors, actors, writers).

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: SERIAL, PRIMARY KEY
    \item \texttt{first\_name}: VARCHAR(50), NOT NULL
    \item \texttt{last\_name}: VARCHAR(50), NOT NULL
    \item \texttt{birth\_year}: INTEGER, NULL, CHECK (birth\_year >= 1800 AND birth\_year <= 2010)
\end{itemize}

\noindent\textbf{Primary key:} Surrogate key \texttt{id}. A surrogate key is preferred because multiple people can share the same name.

\noindent\textbf{Foreign keys:} None.

\subsubsection{Rating}

\noindent\textbf{Purpose:} Stores numerical movie ratings submitted by registered users.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: SERIAL, PRIMARY KEY
    \item \texttt{user\_id}: INTEGER, NOT NULL, FOREIGN KEY
    \item \texttt{movie\_id}: INTEGER, NOT NULL, FOREIGN KEY
    \item \texttt{score}: INTEGER, NOT NULL, CHECK (score >= 1 AND score <= 10)
    \item \texttt{created\_at}: TIMESTAMP, NOT NULL, DEFAULT CURRENT\_TIMESTAMP
\end{itemize}

\noindent\textbf{Primary key:} Surrogate key \texttt{id}. Although the combination (\texttt{user\_id}, \texttt{movie\_id}) could serve as a natural primary key, a surrogate key is more efficient for indexing.

\noindent\textbf{Foreign keys:}
\begin{itemize}
    \item \texttt{user\_id} REFERENCES \texttt{User(id)}: ON DELETE CASCADE
    \item \texttt{movie\_id} REFERENCES \texttt{Movie(id)}: ON DELETE CASCADE
\end{itemize}

\subsubsection{Review}

\noindent\textbf{Purpose:} Stores text reviews of movies written by registered users.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: SERIAL, PRIMARY KEY
    \item \texttt{user\_id}: INTEGER, NOT NULL, FOREIGN KEY
    \item \texttt{movie\_id}: INTEGER, NOT NULL, FOREIGN KEY
    \item \texttt{title}: VARCHAR(200), NOT NULL
    \item \texttt{content}: TEXT, NOT NULL, CHECK (LENGTH(content) >= 50)
    \item \texttt{created\_at}: TIMESTAMP, NOT NULL, DEFAULT CURRENT\_TIMESTAMP
\end{itemize}

\noindent\textbf{Primary key:} Surrogate key \texttt{id}. Each review is a unique record identified by a numeric ID.

\noindent\textbf{Foreign keys:}
\begin{itemize}
    \item \texttt{user\_id} REFERENCES \texttt{User(id)}: ON DELETE CASCADE
    \item \texttt{movie\_id} REFERENCES \texttt{Movie(id)}: ON DELETE CASCADE
\end{itemize}

\subsubsection{Watchlist}

\noindent\textbf{Purpose:} Stores movie lists created by registered users.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: SERIAL, PRIMARY KEY
    \item \texttt{user\_id}: INTEGER, NOT NULL, FOREIGN KEY
    \item \texttt{name}: VARCHAR(100), NOT NULL
    \item \texttt{description}: TEXT, NULL
    \item \texttt{is\_public}: BOOLEAN, NOT NULL, DEFAULT TRUE
    \item \texttt{created\_at}: TIMESTAMP, NOT NULL, DEFAULT CURRENT\_TIMESTAMP
\end{itemize}

\noindent\textbf{Primary key:} Surrogate key \texttt{id}. User can have multiple watchlists, so surrogate key works as unique identificator.

\noindent\textbf{Foreign keys:}
\begin{itemize}
    \item \texttt{user\_id} REFERENCES \texttt{User(id)}: ON DELETE CASCADE
\end{itemize}

\subsubsection{Genre}

\noindent\textbf{Purpose:} Lookup table for genre names.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: SERIAL, PRIMARY KEY
    \item \texttt{name}: VARCHAR(30), NOT NULL, UNIQUE
\end{itemize}

\noindent\textbf{Primary key:} Surrogate key \texttt{id}. Although genre name is unique and could serve as a primary key, a surrogate key works better in a junction table which connects movies and genres together.
\noindent\textbf{Foreign keys:} None.

\subsubsection{MovieGenre}

\noindent\textbf{Purpose:} Junction table representing a M:N relationship between movies and genres.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{movie\_id}: INTEGER, NOT NULL, FOREIGN KEY
    \item \texttt{genre\_id}: INTEGER, NOT NULL, FOREIGN KEY
\end{itemize}

\noindent\textbf{Primary key:} Composite key (\texttt{movie\_id}, \texttt{genre\_id}). 

\noindent\textbf{Foreign keys:}
\begin{itemize}
    \item \texttt{movie\_id} REFERENCES \texttt{Movie(id)}: ON DELETE CASCADE
    \item \texttt{genre\_id} REFERENCES \texttt{Genre(id)}: ON DELETE CASCADE
\end{itemize}

\subsubsection{MoviePerson}

\noindent\textbf{Purpose:} Junction table representing a M:N relationship between movies and directors, actors and writers.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{movie\_id}: INTEGER, NOT NULL, FOREIGN KEY
    \item \texttt{person\_id}: INTEGER, NOT NULL, FOREIGN KEY
    \item \texttt{role}: ENUM('actor', 'director', 'writer'), NOT NULL
\end{itemize}

\noindent\textbf{Primary key:} Composite key (\texttt{movie\_id}, \texttt{person\_id}, \texttt{role}). Role is included in composite key as the same person can be a director and an actor in the same movie.

\noindent\textbf{Foreign keys:}
\begin{itemize}
    \item \texttt{movie\_id} REFERENCES \texttt{Movie(id)}: ON DELETE CASCADE
    \item \texttt{person\_id} REFERENCES \texttt{Person(id)}: ON DELETE CASCADE
\end{itemize}

\subsubsection{WatchlistMovie}

\noindent\textbf{Purpose:} Junction table representing a M:N relationship
between watchlists and movies. Allows a single watchlist to contain
multiple movies, and a single movie to appear in multiple watchlists.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{watchlist\_id}: INTEGER, NOT NULL, FOREIGN KEY
    \item \texttt{movie\_id}: INTEGER, NOT NULL, FOREIGN KEY
    \item \texttt{added\_at}: TIMESTAMP, NOT NULL, DEFAULT CURRENT\_TIMESTAMP
\end{itemize}

\noindent\textbf{Primary key:}
\noindent\textbf{Foreign keys:}
\begin{itemize}
    \item \texttt{watchlist\_id} REFERENCES \texttt{Watchlist(id)}:
          ON DELETE CASCADE — removing a watchlist removes all its
          movie entries
    \item \texttt{movie\_id} REFERENCES \texttt{Movie(id)}:
          ON DELETE CASCADE — removing a movie removes it from all
          watchlists
\end{itemize}


\subsection{Table relationships}

\subsubsection{User and Rating}
\begin{itemize}
    \item Relationship type: 1:N - User can rate multiple movies. A primary key from table User can appear multiple times as foreign key in table Rating.
    \item Cardinality: 1 (User) 0..* (Rating) - User can rate 0 or more movies.
    \item Mandatory: \texttt{user_id} NOT NULL - Rating must belong to a user.
    \item Deleting: \texttt{user_id}: CASCADE - Deleting a user removes all his ratings. %mozno dat SET NULL
\end{itemize}

\subsubsection{User and Review}
\begin{itemize}
    \item Relationship type: 1:N - User can review multiple movies. A primary key from table User can appear multiple times as foreign key in table Review.
    \item Cardinality: 1 (User) 0..* (Review) - User can review 0 or more movies.
    \item Mandatory: \texttt{user_id} NOT NULL - Review must belong to a user.
    \item Deleting: \texttt{user_id}: CASCADE - Deleting a user removes all his reviews. 
\end{itemize}

\subsubsection{User and Watchlist}
\begin{itemize}
    \item Relationship type: M:N - User can create multiple watchlists and share them among friends. Watchlist can have multiple owners. This relationship is realized by using junction table WatchlistUser.
    \item Cardinality: 1 (User) 0..* (WatchlistUser), 1..* (WatchlistUser) 1 (Watchlist), - User can own 0 or more watchlists. Watchlist has to have at least one owner.
    \item Mandatory: \texttt{user_id} NOT NULL, \texttt{watchlist_id} NOT NULL. A row with one of these values as NULL would not make sense.
    \item Deleting: \texttt{user_id}: CASCADE - Deleting a user removes all his relationships with watchlists. \texttt{watchlist_id}: CASCADE - Deleting a watchlist removes all relationships with users.
\end{itemize}

\subsubsection{Watchlist and Movie}
\begin{itemize}
    \item Relationship type: M:N - Watchlist can have multiple movies. A movie can appear in multiple watchlists. This relationship is realized by using junction table WatchlistMovie.
    \item Cardinality: 1 (Watchlist) 0..* (WatchlistMovie), 0..* (WatchlistMovie) 1 (Movie) - Watchlist can consist of 0 or more movies. A movie can appear in 0 or more watchlists.     
    \item Mandatory: \texttt{watchlist_id} NOT NULL, \texttt{movie_id} NOT NULL. A row with one of these values as NULL would not make sense.
    \item Deleting: \texttt{watchlist_id}: CASCADE - Deleting a watchlist removes relationships with movies in that particular watchlist. \texttt{movie_id}: CASCADE - Deleting a movie removes it from every watchlist. 
\end{itemize}
