\section{Specification}
\label{sec:specification}

\subsection{Problem despription}

\subsection{Actors}

\begin{enumerate}
    \item Visitor (unregistered user)
    \begin{itemize}
        \item Can browse movie catalogue
        \item Can search and filter movies in the catalogue
        \item Can read reviews and view ratings
    \end{itemize}
    \item Registered user
    \begin{itemize}
        \item Has all visitor's rights
        \item Can rate movies
        \item Can write reviews
        \item Can create watchlists and share them amongst other users
    \end{itemize}
    \item Moderator
    \item Administrator
    \begin{itemize}
        \item Has all moderator's rights
        \item Can add new movies into database
        \item Can modify movie attributes
    \end{itemize}
\end{enumerate}

\subsection{Key entities}

\begin{enumerate}
    \item Movie
    \item User
    \item Review
    \item Rating
    \item Watchlist
    \item Genre
    \item Person
\end{enumerate}

\subsection{Business rules and restrictions}

\begin{enumerate}
    \item \textbf{Rating uniqueness}
    One user can rate a movie just once. This also includes that only registered users can rate a movie.
    \begin{itemize}
        \item Integrity constraint: UNIQUE constraint on (\texttt{user\_id}, \texttt{movie\_id}) in Rating table
        \item Integrity constraint: NOT NULL foreign key on \texttt{user\_id} in Rating table
        \item Reason: Prevents duplicate ratings that would skew the average score
    \end{itemize}
    \item \textbf{Movie uniqueness}
    Combination of title and release year must be unique for every movie to prevent duplicate entries.
    \begin{itemize}
        \item Integrity constraint: UNIQUE constraint on (\texttt{title}, \texttt{release\_year}) in Movie table
        \item Reason: The same movie with the same release year cannot be added twice
    \end{itemize}
    \item \textbf{Valid release year}
    Movie release year must be realistic.
    \begin{itemize}
        \item Integrity constraint: CHECK constraint (\texttt{release\_year} >= 1888 AND \texttt{release\_year} <= 2100)
        \item Reason: The first movie was filmed in 1888, the upper limit protects against input errors
    \end{itemize}
\end{enumerate}

\section{Use cases}
\label{sec:use_cases}

\subsection{Searching and filtering movies}

Visitor or registered user filters movies based on multiple criteria (genre, release year, rating, director, actors) and sorts them.

\textbf{Main flow:}

\begin{enumerate}
    \item User opens search tab
    \item System displays available filters
    \item User selects and applies filters
    \item User chooses sorting option
    \item System creates SQL query and applies it onto the database
    \item System shows filtered movies
\end{enumerate}

\textbf{Alternate flows:}

\begin{enumerate}
    \item No results
    \begin{itemize}
        \item No movies match the selected filters
        \item System shows ``no matches'' message
        \item User adjusts the filters
    \end{itemize}
\end{enumerate}

\section{Relational Model Design}

\subsection{Overview of tables and attributes}

\subsubsection{User}

\noindent\textbf{Purpose:} Stores information about registered users.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: INTEGER, AUTO\_INCREMENT, PRIMARY KEY
    \item \texttt{first\_name}: VARCHAR(50), NOT NULL
    \item \texttt{last\_name}: VARCHAR(50), NOT NULL
    \item \texttt{username}: VARCHAR(30), NOT NULL, UNIQUE
    \item \texttt{email}: VARCHAR(254), NOT NULL, UNIQUE
    \item \texttt{password\_hash}: VARCHAR(255), NOT NULL
    \item \texttt{role}: ENUM('user', 'admin'), NOT NULL, DEFAULT 'user'
    \item \texttt{created\_at}: TIMESTAMP, NOT NULL, DEFAULT CURRENT\_TIMESTAMP
\end{itemize}

\noindent\textbf{Primary key:} Artificial key \texttt{id} (AUTO\_INCREMENT). An artificial key is preferred over username or email because both, although unique, might be changed in the future.

\noindent\textbf{Foreign keys:} None.

\subsubsection{Movie}

\noindent\textbf{Purpose:} Stores information about movies.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: INTEGER, AUTO\_INCREMENT, PRIMARY KEY
    \item \texttt{title}: VARCHAR(200), NOT NULL
    \item \texttt{release\_year}: INTEGER, NOT NULL, CHECK (release\_year >= 1888 AND release\_year <= 2030)
    \item \texttt{runtime\_minutes}: INTEGER, CHECK (runtime\_minutes > 0)
    \item \texttt{average\_rating}: DECIMAL(3,1), DEFAULT NULL, CHECK (average\_rating >= 0 AND average\_rating <= 10)
    \item \texttt{total\_ratings}: INTEGER, NOT NULL, CHECK (total\_ratings >= 0), DEFAULT 0
    \item \texttt{created\_at}: TIMESTAMP, NOT NULL, DEFAULT CURRENT\_TIMESTAMP
\end{itemize}

\noindent\textbf{Primary key:} Artificial key \texttt{id} (AUTO\_INCREMENT). An artificial key is preferred over the combination of title and release year, because title can be very long and id is more efficient for indexing.

\noindent\textbf{Foreign keys:} None.

\subsubsection{Person}

\noindent\textbf{Purpose:} Stores information about people involved in movies (directors, actors, writers).

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: INTEGER, AUTO\_INCREMENT, PRIMARY KEY
    \item \texttt{first\_name}: VARCHAR(50), NOT NULL
    \item \texttt{last\_name}: VARCHAR(50), NOT NULL
    \item \texttt{birth\_year}: INTEGER, NULL, CHECK (birth\_year >= 1800 AND birth\_year <= 2010)
\end{itemize}

\noindent\textbf{Primary key:} Artificial key \texttt{id} (AUTO\_INCREMENT). An artificial key is preferred because multiple people can share the same name.

\noindent\textbf{Foreign keys:} None.

\subsubsection{Rating}

\noindent\textbf{Purpose:} Stores numerical movie ratings submitted by registered users.

\noindent\textbf{Attributes:}
\begin{itemize}
    \item \texttt{id}: INTEGER, AUTO\_INCREMENT, PRIMARY KEY
    \item \texttt{user\_id}: INTEGER, NOT NULL, FOREIGN KEY, REFERENCES User(id)
    \item \texttt{movie\_id}: INTEGER, NOT NULL, FOREIGN KEY, REFERENCES Movie(id)
    \item \texttt{score}: INTEGER, NOT NULL, CHECK (score >= 1 AND score <= 10)
    \item \texttt{created\_at}: TIMESTAMP, NOT NULL, DEFAULT CURRENT\_TIMESTAMP
\end{itemize}

\noindent\textbf{Primary key:} Artificial key \texttt{id} (AUTO\_INCREMENT). Although the combination (\texttt{user\_id}, \texttt{movie\_id}) could serve as a natural primary key, an artificial key is more efficient for indexing.

\noindent\textbf{Foreign keys:}
\begin{itemize}
    \item \texttt{user\_id} REFERENCES \texttt{User(id)}: ON DELETE CASCADE
    \item \texttt{movie\_id} REFERENCES \texttt{Movie(id)}: ON DELETE CASCADE
\end{itemize}

\section{Implementation}
\label{sec:implementation}

The template lives in a simple directory structure with localization files
separated from the main document. Switching language requires changing one
line:

\begin{lstlisting}[language=TeX, caption={Switching language in the preamble}]
% English
\input{lang/en}

% Slovak
\input{lang/sk}
\end{lstlisting}

For SQL or other languages, change the \texttt{language} parameter:

\begin{lstlisting}[language=SQL, caption={Example SQL query}]
SELECT s.name, COUNT(*) AS submission_count
FROM students s
JOIN submissions sub ON s.id = sub.student_id
GROUP BY s.name
ORDER BY submission_count DESC;
\end{lstlisting}

References to tables, figures, and sections are automatic. For example,
this is Section~\ref{sec:implementation}, and the faculty logo is shown in
Figure~\ref{fig:logo} in Attachment~\ref{sec:attachment-a1}.
If you add or reorder content, the numbers update on the next build.

The database fundamentals referenced throughout the course are covered
in \cite{elmasri2016}. For PostgreSQL-specific syntax, see
\cite{postgresql-docs}.

\section{Challenges}
\label{sec:challenges}

The main challenge was keeping the template simple enough that a student
seeing \LaTeX{} for the first time would not immediately close the file.
We solved this by limiting the preamble to essential packages and
putting all examples in one place --- this document.

We also learned that BibTeX interprets \texttt{@} signs in comments as
entry types, which caused confusing errors. Small things like this are
easy to fix once you know about them, but hard to debug the first time.
